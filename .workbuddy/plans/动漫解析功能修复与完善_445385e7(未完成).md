## 用户需求分析

**核心问题**: 用户的动漫规则仓库使用 `api=1` 的简化 CSS Selector 格式,但项目原本期望 `api=2` 的 XPath 格式,导致动漫列表和详情页完全无法加载。

**用户需求**:

1. 让动漫列表能够正确加载显示
2. 让动漫详情页能够正确解析剧集信息
3. 兼容用户的规则仓库格式 (https://github.com/jipika/WallHaven-Profiles)
4. 兼容本地配置文件的 XPath 格式 (AnimeProfiles.sample.json)
5. 仔细检查并修复所有相关问题

**补充发现**:

- 用户规则仓库使用 CSS Selector (api=1)
- 本地配置文件 AnimeProfiles.sample.json 使用 XPath 格式
- 项目需要同时支持两种格式
- 已有代码已实现基础的双格式支持,但需要仔细验证和完善

## 产品概述

修复并完善 WallHaven 应用的动漫解析功能,使其能够正确加载和显示动漫内容列表及详情页,同时支持两种规则格式。

## 核心功能

1. **双格式解析引擎**: 支持 api=1 (CSS Selector) 和 api=2 (XPath) 两种规则格式
2. **智能路由**: 根据规则类型自动选择解析方式
3. **兼容性保障**: 同时兼容用户规则仓库和本地配置文件
4. **错误处理**: 完善的错误处理和降级机制
5. **测试验证**: 完整的测试用例覆盖

## 技术栈

- **语言**: Swift
- **解析库**: SwiftSoup (HTML 解析)
- **架构**: MVVM + Actor 并发模型
- **数据格式**: JSON 规则配置

### 关键技术细节

**XPath 转 CSS Selector 模式**:

```
//div[contains(@class,'item')] → div.item 或 div[class*="item"]
.//a/@href → a[href]
//div[@id='main']/ul/li → div#main ul li
```

**占位符处理**:

```
{keyword} → URL 编码的关键词
@keyword → 同上 (本地配置格式)
{page} → 页码
```

**相对路径转换**:

```
/detail/123 → https://www.agedm.io/detail/123
//cdn.com/img.jpg → https://cdn.com/img.jpg
```

### 数据流

```
用户搜索请求
    ↓
AnimeViewModel.search()
    ↓
AnimeParser.search() → 规则遍历 → 格式检测
    ↓
parseSearchResultsV1/V2() → CSS/XPath 解析
    ↓
返回 AnimeSearchResult[]
    ↓
显示列表
    ↓
用户点击详情
    ↓
AnimeParser.fetchDetail()
    ↓
parseDetailV1/V2() → 剧集解析
    ↓
返回 AnimeDetail
    ↓
显示详情页
```
